# 快速部署

## 最簡單流程

按：

```
Ctrl + Shift + B
```

之後照問題回答就好：

```text
1. 要不要先儲存全部檔案，預設 Y
2. 要 push 測試版，還是 deploy 正式版
3. 要更新 patch / minor / major / none，預設 none
4. 要不要寫入 README 版本日誌
   - none / patch 預設 N
   - minor / major 預設 Y
5. 如果 README 選 Y，才會問版本日誌要寫什麼
6. 要不要順便 push GitHub
```

建議回答：

```text
小修正：
  patch → push → README 選 N

新增功能：
  minor → push → README 選 Y

大改版：
  major → push → README 選 Y

確認要上正式版：
  none → deploy → README 選 Y
```

none 代表不升 major / minor / patch，只更新時間戳；適合重跑 deploy 或不想讓版本號一直增加時使用。

GitHub 那題：

```text
留空：不 commit、不 push GitHub
有輸入：自動 git add .、git commit、git push
```

## 手動執行

```powershell
.\push.ps1
```

它跟 `Ctrl + Shift + B` 是同一套流程。

正式版

Ctrl + Shift + B  
→ 依問題選 deploy  
→ clasp push  
→ 建立 Apps Script version  
→ 更新正式 deployment  
→ 可選擇是否寫 README 與 GitHub

# 版本規則

格式：

```text
vMajor.Minor.Patch-YYYYMMDDHHmm
```

範例：

```text
v1.1.5-202605262315
```

說明：

- Major → 大改版
- Minor → 新功能
- Patch → 修正
- 時間 → push 時間

版本階段：

```text
0.x.x → 開發中
1.x.x 以上 → 穩定版
```

# 敷料主檔建立思路整理

## 一、核心概念

掃描到的完整條碼（UDI / GS1）不是產品本身。

完整條碼裡面包含：

* GTIN（產品識別）
* LOT（批號）
* EXP（效期）

真正代表「敷料種類」的是：

```text
GTIN
```

也就是：

```text
(01) 後面的值
```

例如：

```text
(01)14900070187522
```

GTIN：

```text
14900070187522
```

---

## 二、為什麼不能直接用完整條碼當產品

因為：

同一個敷料：

* 不同批號
* 不同效期
* 不同單包

都會有不同完整條碼。

所以：

```text
完整條碼 ≠ 敷料種類
```

應該：

```text
GTIN → 對應敷料
```

---

## 三、目前系統第一步

先建立：

```text
敷料主檔
```

目的：

建立：

```text
GTIN → 敷料資訊
```

對照表。

之後掃描：

```text
raw barcode
↓
解析 UDI
↓
提取 GTIN
↓
查敷料主檔
↓
帶出敷料名稱
```

---

## 四、目前規劃的敷料主檔欄位

| 欄位名稱   | 用途                       |
| ------ | ------------------------ |
| 院內碼    | 主識別 ID                   |
| 敷料名稱   | 顯示名稱                     |
| 規格描述   | 10×10、厚、有邊               |
| 敷料分類   | foam、silver、hydrocolloid |
| 包裝GTIN | 外盒條碼                     |
| 包裝內數量  | 一盒幾片                     |
| 單包GTIN | 單片 UDI GTIN              |
| 條碼類型   | UDI／商品碼                  |
| 啟用狀態   | 啟用／試用／停用                 |
